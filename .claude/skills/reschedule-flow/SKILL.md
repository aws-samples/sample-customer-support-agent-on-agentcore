# 调课操作流程

## 触发条件
用户表达调课意图，如：
- "我想调课"
- "能把课改到其他时间吗"
- "周六的课想换到周日"
- "那节课时间有冲突，能改吗"

## 标准流程

### Step 1: 查询原课程
调用 `get_booking_records` 查询用户要调整的课程，获取：
- `booking_id`: 预约记录ID
- 原课程时间
- 原老师信息
- 学生信息

### Step 2: 确定新时间
- 如用户指定了新时间的具体时刻，使用用户指定的时间
- 如用户只指定新日期（如"改到周日"），**默认继承原课程的时间点**

**示例**：
- 原课程：周六 10:00
- 用户说"改到周日"
- 默认新时间：周日 10:00

### Step 3: 检查可用性
调用 `get_available_slots` 检查：
- 原老师在新时间是否有空（优先）
- 如原老师无空，展示其他老师的可用时段

### Step 4: 向用户确认
展示调课信息并请求确认：
- 原课程时间
- 新课程时间
- 老师（是否变更）
- 学生

**示例回复**：
```
好的，为您确认调课信息：

原课程：2024-03-15 周六 10:00 - 王老师 - 小明
调整为：2024-03-17 周日 10:00 - 王老师 - 小明

请确认是否调课？
```

### Step 5: 执行调课
用户确认后，调用 `reschedule_class` 完成调课：
- `parent_id`: 家长用户ID
- `booking_id`: 预约记录ID
- `new_time`: 新时间（格式 YYYY-MM-DD HH:MM）
- `new_tutor_id`: 新老师ID（如需更换老师）

### Step 6: 反馈结果
调课成功后告知用户新的课程安排。

## 注意事项

1. **提前2小时**：调课需在原课程开始前2小时操作
2. **优先原老师**：优先保持原老师，除非用户指定更换
3. **继承时间点**：如用户只说日期不说时间，继承原课程时间点
4. **单次操作**：每次只处理一节课的调整
5. **小班课**：小班课调课会消耗调课机会，需提醒用户

## 特殊情况处理

### 原老师新时间无空
```
抱歉，王老师在周日10:00没有空余时段。

您可以选择：
1. 周日14:00 - 王老师（同老师，不同时间）
2. 周日10:00 - 李老师（同时间，不同老师）

请问您想选择哪个？
```

### 时间过期
```
抱歉，这节课距离上课时间不足2小时，无法调课。
如需调整，请联系顾问协助处理。
```

### 用户想同时换时间和老师
完全支持，在 `reschedule_class` 中同时提供 `new_time` 和 `new_tutor_id`。

### 批量调课请求
```
我只能帮您调整单节课程。如果您想批量调整多节课程，
请联系顾问协助处理。
```
